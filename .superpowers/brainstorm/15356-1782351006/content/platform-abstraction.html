平台抽象层设计

渐进式抽取，统一管理平台差异

当前问题

20+ 处 process.platform 分散在 10+ 个文件中，类型分散、逻辑重复：

  • 路径大小写比较 → 5 处重复 toLowerCase()
  • 进程终止方式 → 3 处重复 proc.kill() vs SIGTERM
  • CLI 工具检测 → 2 处重复 where vs which
  • 系统目录屏蔽 → 3 处重复 Windows 路径前缀判断
  • WSL 检测 → 孤立逻辑

抽取优先级（3 批渐进）

按影响面排序，每批可独立验证
📦 Batch 1 — 路径操作（5 处重复，最高优先）
   src/main/ipc/utils.ts    → pathsEqual / isSystemPath
   src/main/ipc-handlers.ts → pathsEqual (×3)
   src/main/ipc/fs.ts       → isWithinParent
   src/main/ipc/git.ts      → pathsEqual
   src/main/scope-guard.ts  → pathsEqual / isWithinParent

📦 Batch 2 — 进程管理（4 处，影响 Agent 执行）
   src/main/adapters/base.ts     → killProcess (×2) + getShellConfig
   src/main/adapters/mindmap-adapter.ts → getShellConfig
   src/main/settings.ts          → whichCommand (×2)

📦 Batch 3 — 剩余零散（3 处，低优先）
   src/main/index.ts             → isMac (菜单构建)
   src/main/scope-guard.ts       → isWsl (watcher config)
   src/main/project-scanner/     → isSystemPath

实现原则

单例 + 懒初始化：getPlatformProvider() 首次调用时根据 process.platform 创建对应实例，进程生命周期内复用。

不改变外部行为：每批抽取后，原有函数签名和行为完全不变，只是内部实现委托给 Provider。可逐文件迁移验证。

可测试：注入 mock Provider 即可在任意平台测试所有平台分支逻辑，不再依赖真实 OS 环境。

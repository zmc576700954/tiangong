各方案 Windows ARM64 支持现状

npm 原生包覆盖对比
数据库 类型 win-x64 win-arm64 mac-arm64 linux-arm64
@libsql/client Rust native ✅ ❌ ✅ ✅
better-sqlite3 C++ native ✅ ✅ ✅ ✅
sql.js WASM ✅ ✅ ✅ ✅

方案 A：替换为 better-sqlite3（推荐）

优势

  • 全架构原生支持，win32-arm64 有 prebuild
  • 性能最优（同步 API，比 libsql 更快）
  • prebuild-install 自动下载，CI 无需额外配置
  • 成熟项目，Electron 生态首选嵌入式 DB
  • 无需 x64 模拟层或 HTTP 降级

劣势

  • 需要重写 database.ts 数据访问层
  • API 从 libsql 切换到 better-sqlite3（语法差异）
  • 失去 libsql 特有功能（嵌入式副本、HTTP 同步）

方案 B：DatabaseProvider 抽象层（双引擎）

优势

  • 保留 libsql，新增 better-sqlite3 作为 ARM64 fallback
  • 按平台自动选择最优引擎
  • 渐进式迁移，不破坏现有功能

劣势

  • 维护两套数据库驱动，复杂度高
  • 两套 SQL 行为可能有微妙差异
  • 包体积增大（两个原生库）

方案 C：sql.js 统一方案

优势

  • 零原生依赖，全平台统一
  • 无需任何平台适配

劣势

  • 全库加载到内存，大库 OOM 风险
  • 性能远差于原生方案
  • sql.js 官方不推荐 Electron 使用

💡 推荐：方案 A — 替换为 better-sqlite3

很多开源项目不做降级/模拟层，正是因为选择了一个全架构覆盖的数据库。better-sqlite3 是 Electron 生态最成熟的嵌入式数据库，6 个目标架构全部有 prebuild，无需任何兼容层或降级策略。

BizGraph 当前未使用 libsql 的特有功能（嵌入式副本、Turso 同步），仅用其 SQLite 子集，迁移成本低。

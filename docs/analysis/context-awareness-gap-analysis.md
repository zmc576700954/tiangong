# BizGraph 上下文感知能力差距分析

> 对比对象：Cursor IDE、ZedAgent 等现代 AI 辅助 IDE
> 分析日期：2026/06/08

---

## 一、Cursor / ZedAgent 的上下文感知机制

### 1.1 多层级上下文体系

现代 IDE 的 AI 助手通常采用**多层级上下文体系**，按优先级和动态性分层：

```
Layer 1: 即时上下文（最高优先级，动态更新）
  - 当前光标位置 / 选中代码
  - 当前打开的文件内容
  - 最近编辑的文件
  - 当前诊断信息（错误/警告）

Layer 2: 会话上下文（中优先级，会话级维护）
  - 对话历史
  - 已引用/已修改的文件
  - 用户确认的假设和决策

Layer 3: 工作区上下文（低优先级，相对稳定）
  - 项目结构 / 目录树
  - Import/依赖关系图
  - 类型定义和接口
  - 测试文件关联

Layer 4: 全局上下文（最低优先级，预计算）
  - 语义代码索引（向量嵌入）
  - 符号索引（函数/类/变量定义）
  - 代码摘要和社区聚类
```

### 1.2 文件相关性评分（File Relevance Scoring）

Cursor 和 Zed 使用多维评分模型来确定哪些文件与当前任务最相关：

| 维度 | 权重 | 说明 |
|------|------|------|
| **语义相似度** | 高 | 用户查询与文件内容的向量相似度（通过 code embeddings） |
| **结构 proximity** | 高 | 在 import 图中距离当前文件的跳数 |
| **编辑时间衰减** | 中 | 最近编辑的文件获得更高权重（时间衰减函数） |
| **符号引用** | 高 | 当前文件中引用的符号定义位置 |
| **同模块聚类** | 中 | 同一目录/模块下的文件 |
| **测试关联** | 中 | 与被修改文件对应的测试文件 |
| **用户显式引用** | 最高 | @-mentions 或显式添加的文件 |

**评分公式（简化）**：
```
relevance_score = α·semantic_sim + β·(1/import_distance) + γ·time_decay + δ·symbol_ref + ε·module_cluster + ζ·test_assoc + η·explicit_ref
```

### 1.3 关键词提取与意图分析

Cursor 的 Composer / Agent 模式在接收到用户自然语言需求时，会执行以下分析：

1. **意图分类**：识别是 "add feature"、"fix bug"、"refactor"、"explain"、"test" 中的哪一类
2. **实体抽取**：从需求中提取关键实体（类名、函数名、文件路径、API 名等）
3. **动作分解**：将复杂需求拆分为可执行的子任务（如：先读取 X 文件，再修改 Y 函数，最后添加测试）
4. **搜索查询生成**：将实体转化为代码库搜索查询（支持模糊匹配、同义词扩展）

### 1.4 Prompt 拼接策略

现代 IDE 的 Prompt 工程遵循 "least-to-most" 原则：

```
System Prompt (固定)
  ├── 角色定义（你是一名专家级软件工程师）
  ├── 工具描述（可使用的文件操作工具）
  ├── 输出格式要求
  └── 约束条件

Dynamic Context (动态构建)
  ├── 当前文件片段（光标前后 N 行）
  ├── 相关文件内容（按相关性评分排序，截断到 token 预算）
  ├── 类型定义（当前符号的类型信息）
  ├── Import 依赖（当前文件的 import 列表）
  ├── 最近修改的文件 diff
  └── 诊断信息（当前文件的 TypeScript 错误）

User Prompt (用户输入)
  └── 原始需求 + 意图分类标签
```

### 1.5 自动文件发现机制

Cursor Agent 模式的自动文件发现流程：

```
1. 接收用户请求
   ↓
2. 意图分析 → 确定任务类型（feature/fix/refactor/test）
   ↓
3. 关键词提取 → 生成搜索查询
   ↓
4. 并行搜索：
   ├── 语义搜索（向量相似度 top-K）
   ├── 符号搜索（精确匹配类名/函数名）
   ├── 文件名搜索（模糊匹配文件路径）
   └── 最近编辑文件（时间衰减排序）
   ↓
5. 相关性评分 → 排序合并结果
   ↓
6. 读取高相关性文件内容（按需读取，不是一次性全读）
   ↓
7. 生成执行计划（多步骤工具调用序列）
   ↓
8. 逐步执行，根据中间结果调整上下文
```

| **项目结构扫描** | `project-scanner/` 7步流水线 | ✅ 框架检测 + 目录扫描 |

| **复杂度分类** | `complexity-classifier.ts` 4级分类 | ✅ 简单的规则分类 |

| **上下文引用** | `ContextRef` 支持 node/file/text | ✅ 基础引用机制 |

### 2.2 关键缺失的能力 ❌

| 缺失能力 | 影响 | 优先级 |
|----------|------|--------|
| **AST 解析** | 无法理解代码结构，只能正则匹配 | 🔴 P0 |
| **语义搜索/向量嵌入** | 无法找到语义相关但字面不匹配的文件 | 🔴 P0 |
| **Import/依赖图** | 无法追踪 "谁调用了谁" | 🔴 P0 |
| **类型信息提取** | 无法理解 TypeScript 接口和类型 | 🟡 P1 |
| **智能文件发现** | 用户必须手动指定文件，不会自动找 | 🔴 P0 |
| **关键词语义分析** | 只有 split+去重，无 NLP | 🟡 P1 |
| **编辑器状态感知** | 不知道用户在看什么 | 🟡 P1 |
| **代码变更 diff** | 没有最近修改的上下文 | 🟡 P1 |
| **LSP 集成** | 没有语言服务器支持 | 🟢 P2 |
| **动态上下文更新** | 上下文是静态快照，不会根据中间结果调整 | 🟢 P2 |

## 三、核心差距详细分析

### 3.1 差距一：无代码语义理解（AST 缺失）

**Cursor/Zed 的做法**：
- 使用 Tree-sitter / TypeScript Compiler API 解析代码
- 构建符号索引：类、函数、变量、接口的定义和引用位置
- 支持 goto-definition、find-references
- 理解作用域和类型层次

**BizGraph 的现状**：
- 所有代码分析基于正则表达式
- `entity-parser.ts` 用 `class\s+(\w+)` 匹配类名
- `route-parser.ts` 用正则匹配路由定义
- **无法处理**：嵌套类、泛型、装饰器、复杂类型、重载函数

**影响场景**：
- 用户说 "修改 UserService 的 create 方法" → BizGraph 无法精确定位到 `UserService` 类中的 `create` 方法
- 用户说 "给所有 API 添加日志中间件" → BizGraph 无法找到所有 API 端点定义

### 3.2 差距二：无语义搜索能力

**Cursor/Zed 的做法**：
- 使用代码嵌入模型（如 OpenAI text-embedding-3 或专门的 code embedding）
- 将代码片段编码为向量，存储在本地向量数据库
- 用户查询也编码为向量，通过余弦相似度找到最相关的代码
- 支持同义词和语义匹配："用户认证" 可以匹配到 "auth/login/user verification"

**BizGraph 的现状**：
- `searchFilesRecursive` 只按文件名做子字符串匹配
- `directRetrieve` 用节点标题关键词在 `src/` 目录中搜索
- 关键词提取：`extractTerms` 只是 split + filter length > 1
- **完全依赖字面匹配**

**影响场景**：
- 用户说 "实现购物车功能" → 搜索 "购物车" 找不到 `cart.service.ts`
- 用户说 "修复内存泄漏" → 无法找到涉及资源释放的代码

### 3.3 差距三：无依赖关系图

**Cursor/Zed 的做法**：
- 解析所有文件的 import/export 语句
- 构建有向图：模块 A → 导入 → 模块 B
- 查询时沿着依赖边传播相关性
- 修改一个文件时，自动提示需要更新依赖它的文件

**BizGraph 的现状**：
- `project-analyzer.ts` 的依赖检测是字符串匹配：
  - "Module name mentioned in another module's description"
  - "Shared feature names (>=2 overlaps)"
- **没有真正的 import 图分析**

**影响场景**：
- 修改 `User` 接口后，不知道哪些文件引用了它
- 重构一个函数后，不知道哪些调用点需要更新

### 3.4 差距四：Prompt 拼接过于静态

**Cursor/Zed 的做法**：
- 动态构建上下文：根据任务类型选择不同上下文组合
- 自适应截断：保留最相关的代码块，而不是简单截断前 N 行
- 中间结果反馈：Agent 执行过程中，根据已读取的文件内容调整下一步搜索

**BizGraph 的现状**：
- `buildScopePrompt` 是静态模板 + 固定字段填充
- `ContextResolver` 读取文件时只取前 100 行
- `directRetrieve` 读取相关文件时只取前 3000 字符
- **没有根据内容重要性做智能选择**

**影响场景**：
- 一个大文件的实现逻辑在 200-500 行，但只读了前 100 行（可能全是 import 和注释）
- 相关文件有 10 个，但只读了其中几个，因为 token 预算被不重要的内容占用了

### 3.5 差距五：无智能意图分析

**Cursor/Zed 的做法**：
- 使用 LLM 或专门的分类模型分析用户意图
- 将自然语言转化为结构化查询：{"action": "modify", "target": "UserService.create", "properties": ["add logging"]}
- 根据意图类型选择不同的工具链和上下文策略

**BizGraph 的现状**：
- `complexity-classifier.ts` 用正则匹配分类复杂度（global/simple/moderate/complex）
- 没有从用户输入中提取具体实体（类名、方法名、文件路径）
- 没有将需求转化为可执行的搜索查询

**影响场景**：
- 用户说 "把登录改成 JWT" → 系统不知道 "登录" 对应哪些代码，"JWT" 需要引入什么库
- 用户说 "优化这个查询" → 系统不知道 "这个查询" 指的是哪个

## 四、实施计划概览

基于差距分析，制定以下**四阶段实施计划**：

### 阶段一：代码语义索引（P0 - 基础设施）
构建代码的 AST 解析、符号索引和 import 依赖图，让系统真正"理解"代码结构。

### 阶段二：语义搜索与文件发现（P0 - 核心能力）
实现基于代码嵌入的语义搜索，让系统能从自然语言需求中找到相关代码。

### 阶段三：智能上下文组装（P1 - 体验提升）
重构 Prompt 构建逻辑，基于代码重要性、依赖关系和任务类型动态组装上下文。

### 阶段四：意图分析与 Agent 规划（P1 - 智能化）
实现用户需求的意图分析、实体抽取和自动执行规划。

**详细实施计划见：`docs/superpowers/plans/2026-06-08-context-awareness.md`**
